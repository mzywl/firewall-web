## 1. 硬铁律清单 (设计底线, 任何步骤不能违反)

| 铁律 | 出处 | 校验手段 |
|---|---|---|
| **防火墙只认当前进到它接口里的数据包** | 重构.md §2 步骤 B, user 2026-06-19 强调 | chain_planner 对 boundary 上下游 fw 用不同 src 视角 |
| **策略匹配在 SNAT 转换前** | 重构.md §7.2, user 2026-06-19 强调 | chain_planner 边界墙自己的 sp 用原始 src, 不替换 |
| **(direction, src 归属) 二元判定 fw 上下游** | SKILL 坑点 33, 2026-06-20 | chain_planner 内部 helper, 单维度判定禁止 |
| **PushLog 独立事务 (主事务回滚不擦日志)** | 重构.md §8.1, "研发红线" | push_log_writer.py 用独立 SessionLocal + 立即 commit |
| **倒序串行下发** | 重构.md §4, 防止目的端墙未配置时被 SNAT 流量打穿 | push_pipeline 显式 reversed(firewall_ids) |
| **3 mode 查重隔离** | 重构.md §3, 防"既复用又新建"混乱 | push_pipeline 按 mode 分发, 互不干扰 |

## 4. 不做的事 (跟用户偏好对齐)

| 不做项 | 理由 |
|---|---|
| `FirewallZone.connect_region` 字段 (§1) | 当前用 `Firewall.region` / `covered_region` 业务等价, 改造成本高, 收益小 |
| `belong_region` 字段 (§2) | 语义可映射到 `region`, 不需要新字段 |
| `ZoneAccessConfig.need_nat` / `snat_pool` (§3 数据建模) | 当前用 `Firewall.outbound_snat_pool` / `inbound_snat_pool` 已落地, 业务等价 |
| `boundary_source_zone` / `boundary_dest_zone` (§3 数据建模) | 当前 splitter 已能从 `Firewall.local_zone_name` / `external_zone_name` 拿到, 业务等价 |
| 改数据模型 + alembic 迁移 | 用户明确"不动" + 业务等价改造成本高 |
| 改前端 UI | 用户明确"不动" + 后端改完前端自然兼容 (API schema 不破坏) |
