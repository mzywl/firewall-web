# 防火墙策略自动化运维系统——核心后端设计与开发规范文档

本设计文档确立了网络运维自动化系统中“全局业务大区（Region）”**与**“设备本地安全域（Zone）”完全剥离的链式路径计算架构。系统通过动态资产匹配和特征码（Hash）比对，实现了路径预计算、多模式查重以及倒序安全推送。

## 🧭 二、 核心流式处理引擎算法与各流程步骤

本架构弃用了以往依靠 IP 盲撞防火墙的逻辑，升级为“拓扑先行，宏观 Region 寻路，微观 Zone 动态改写，逆序执行下发”的流式自动化下发流水线。

### 【流程 1】工单上传与标准化数据清洗

1. 前端向后端提交 Excel 文件流。
2. 解析引擎逐行读取数据，通过正则表达式、`ipaddress` 库清洗混杂填写的文本。
3. 将类似 `10.1.1.1-10.1.1.5` 的范围或 `192.168.1.1, 192.168.1.2` 等混杂输入，全量统一标准化转换为标准的可迭代 CIDR 集合或单 host IP 字符串。

### 【流程 2】链式寻路算法与 NAT 动态重写

这是核心算法层。针对 Excel 规整后的每一行规则，需要利用 `ipaddress.ip_network` 包含关系计算其完整的动态路由链。

#### 算法伪代码逻辑：

```python
def generate_chain_execution_plan(db, raw_src_ip, raw_dst_ip, service, usage_time):
    # 步骤 A：利用子网归属匹配（子网包含算法），寻找源头和终点所在的归属全局大区
    src_zone_record = db.query(FirewallZone).filter(
        FirewallZone.protected_ips.contains(raw_src_ip) # 严谨做法需要用 ipaddress 匹配底层
    ).first()
    dst_zone_record = db.query(FirewallZone).filter(
        FirewallZone.protected_ips.contains(raw_dst_ip)
    ).first()
    
    if not src_zone_record or not dst_zone_record:
        raise Exception("无法映射源或目的IP的宏观业务大区属性，请先维护安全域资产网段。")
        
    source_region = src_zone_record.connect_region  # 例如: "生产区"
    dest_region = dst_zone_record.connect_region    # 例如: "测试区"
    
    # 建立线性执行计划链条容器
    chain_plan = []
    
    # 状态指针变量：支持链式重写演进
    current_src_ip = raw_src_ip 

    # 步骤 B：判断是否属于跨大区流量（需要检索边界路径矩阵）
    if source_region != dest_region:
        # 1. 精准提取扼守该跨区通道的边界防火墙拓扑及配置
        matrix_route = db.query(ZoneAccessConfig).filter(
            ZoneAccessConfig.source_region == source_region,
            ZoneAccessConfig.dest_region == dest_region
        ).first()
        
        if not matrix_route:
            raise Exception(f"非法越界访问：未配置从 [{source_region}] 去往 [{dest_region}] 的安全访问控制边界矩阵。")
            
        # 2. 首先追加计算“源端本地墙”需要放行的本地策略
        chain_plan.append({
            "firewall_id": src_zone_record.firewall_id,
            "source_ip": current_src_ip, # 沿用原始源IP
            "dest_ip": raw_dst_ip,
            "device_source_zone": src_zone_record.zone_name,
            "device_dest_zone": "untrust", # 去往边界的通用外向安全域
            "source_snat_ip": None
        })
        
        # 3. 追加计算“中转边界墙”需要落地和放行的策略形态（直接从矩阵表中摘取绑定的特定安全域，杜绝同名冲突）
        boundary_plan = {
            "firewall_id": matrix_route.firewall_id,
            "source_ip": current_src_ip, # 仍使用当前的源IP进行边界安全策略和NAT匹配
            "dest_ip": raw_dst_ip,
            "device_source_zone": matrix_route.boundary_source_zone,
            "device_dest_zone": matrix_route.boundary_dest_zone,
            "source_snat_ip": matrix_route.snat_pool if matrix_route.need_nat == 1 else None
        }
        chain_plan.append(boundary_plan)
        
        # 4. 【铁律重写判定】：若边界墙触发了 SNAT 转换机制，后续路径上的源关键特征指针立刻改变！
        if matrix_route.need_nat == 1:
            current_src_ip = matrix_route.snat_pool # 变量特征被永久重写为边界墙的专属SNAT池地址组
            
        # 5. 追加计算“目的端本地墙”需要放行的策略形态（此时源IP已被无缝替换为SNAT后的物理地址段）
        chain_plan.append({
            "firewall_id": dst_zone_record.firewall_id,
            "source_ip": current_src_ip, # 动态变量：可能是真实源IP，也可能是转换后的SNAT池IP
            "dest_ip": raw_dst_ip,
            "device_source_zone": "untrust", # 承接来自边界流量的通用入向域
            "device_dest_zone": dst_zone_record.zone_name,
            "source_snat_ip": None
        })
    else:
        # 属于同区内部同墙或同区内跨设备（东西向微隔离流量放行）
        chain_plan.append({
            "firewall_id": src_zone_record.firewall_id,
            "source_ip": current_src_ip,
            "dest_ip": raw_dst_ip,
            "device_source_zone": src_zone_record.zone_name,
            "device_dest_zone": dst_zone_record.zone_name,
            "source_snat_ip": None
        })
        
    return chain_plan

```

## 🎛️ 三、 三大下发与查重机制（Push Modes）内部深度实现

当操作人员在前端勾选并执行推送工单时，后端需要支持以下 3 种深度隔离的适配计算逻辑：

### 1. 模式一：全新强制推送 (`force_push`)

* **适配机制**：系统完全不与物理防火墙进行线上历史存量数据的比对交互。
* **执行行为**：下发引擎对当前工单策略中的元素采取全额重新创建的原则。为了防止在防火墙上因重名冲突导致下发溃败，系统会自动为所有新创的地址对象（Address）、端口服务组（Service Group）以及安全策略名强制打上当前的工单唯一序列号后缀（例如：`ADDR_10.1.1.5_ORD20260621`）。

### 2. 模式二：对象复用模式 (`reuse_objects`)

* **适配机制**：系统建立连接后，优先将物理防火墙上现存的地址组对象、服务组对象全量反向拉取（Fetch）到内存中。
* **比对逻辑**：系统会将 Excel 中的具体内容（如 `10.1.1.1`）进行标准化清理。通过哈希/内容等价性分析，如果判定设备上已有一个名叫 `Obj_Prod_Web` 的对象也是包含且仅包含 `10.1.1.1`，则系统在后续拼装 CLI 脚本时，**会直接复用已有的对象名** `Obj_Prod_Web`，不再重复创建地址对象。
* **边界动作**：虽然对象复用了，但系统依然会**无条件新创一条独立的安全放行策略规则行**。

### 3. 模式三：策略全量查重模式 (`deduplicate`)

* **适配机制**：企业级最严苛的“静默安全防护比对模式”。系统不仅执行模式二中全套的地址/服务对象复用提取，还会将物理防火墙现存的整条安全策略规则库（Security Policy Rules List）全量抓取。
* **比对核心（4 维度 HASH 特征计算码）**：
系统利用哈希摘要算法，对以下 4 维参数按照名称排序、去空格后生成唯一的 HASH：

$$\text{match\_key} = \text{SHA1}(\text{源对象名称} + \text{目的对象名称} + \text{服务端口对象名称} + \text{有效时间对象名称})$$


* **判定逻辑**：如果该策略行的 `match_key` 与物理设备反馈回来的某个现有策略行的特征码完美重合，并且二者的安全动作（Action）都是 `permit`（放行），则系统在后台将当前策略明细的动作标记为 `reused`（直接复用）。
* **执行行为**：该策略将**不生成任何下发 CLI 脚本**，实现无缝零打扰跳过。

## 🔄 四、 倒序执行下发机制

由于跨区策略涉及多台防火墙联动，并且**后续防火墙必须填边界墙转换后的 SNAT IP**，因此执行层面必须强制推行“逆序串行下发”。

### 🚨 为什么要倒序执行？

如果按照流量方向正序下发（源端墙 ➡️ 边界墙 ➡️ 目的端墙），当边界墙配置完成的瞬间，流量会夹带着被改写后的 SNAT 新 IP 立刻向后奔袭。而此时最后一跳的目的端本地防火墙尚未配置完毕，必然会导致流量在目的端墙被默认阻断策略拦截，从而在监控大屏上触发大面积的安全丢包告警。

* **正解**：先将最后一跳目的端墙放行（此时无流量通过，处于“静默等待状态”），然后配置边界墙（NAT规则生效），最后放起源端墙（整个流量链路瞬间全线贯通）。

## 六、 API 接口设计规范 (FastAPI 路由器)

为了让前端能够流畅地完成“上传 ➡️ 预览 ➡️ 选模式推送 ➡️ 实时看日志”的闭环体验，后端需要暴露以下 4 个核心 API。请让 Claude 严格按照 Pydantic 异步 Schema 来实现。

### 1. 导入与格式化清洗

* **接口**：`POST /api/v1/orders/upload`
* **载荷**：`Multipart/Form-Data` (Excel 文件)
* **内部动作**：接收 Excel 并将其存盘 ➡️ 启动格式化清洗过滤器 ➡️ 批量计算 IP 与服务 ➡️ 将原始数据持久化入 `policy_versions` 表。
* **返回**：返回 `order_id`、`order_no` 以及清洗后的标准化规则预览列表。

### 2. 拓扑寻路与 NAT 链式预分析

* **接口**：`POST /api/v1/orders/{order_id}/analyze`
* **内部动作**：读取工单数据 ➡️ 触发核心**链式寻路算法**（IP ➡️ Zone ➡️ 全局 Region ➡️ 边界墙判定 ➡️ 变量改写） ➡️ 将分裂演变出的多墙物理策略落库至 `policies` 表。
* **返回**：返回完整的**多墙可视化下发计划树（Execution Plan Tree）**，告知前端：“这行 Excel 将在 A 墙生成原 IP 策略，在边界墙生成 NAT 策略，在 B 墙生成 SNAT 后 IP 策略”。

### 3. 异步触发流水线推送

* **接口**：`POST /api/v1/orders/{order_id}/push`
* **载荷**：`{"push_mode": "deduplicate"}` (或 `force_push` / `reuse_objects`)
* **内部动作**：
1. 校验工单当前状态是否为 `pending`。
2. 更改 `orders.status = "processing"`。
3. 为链条中的每台防火墙生成 `PushedPolicySnapshot` 快照记录与随机 `batch_id`。
4. 使用 FastAPI 的 `BackgroundTasks` 或 Celery 异步拉起**倒序下发引擎**，立刻返回 `{"status": "processing", "batch_ids": [...]}` 让前端转圈，绝不阻塞 HTTP 连接。



### 4. 前端高频轮询流式日志

* **接口**：`GET /api/v1/push/logs/{snapshot_id}?last_seq=0`
* **参数**：携带上一次前端已经拿到的最大 `last_seq` 序号。
* **内部动作**：增量查询 `PushLog` 表：`WHERE snapshot_id = :sid AND seq > :last_seq ORDER BY seq ASC`。
* **返回**：返回最新的增量日志数组，前端以此追加到控制台组件中，实现 `~1.5s` 的准实时刷屏。

## 🛠️ 七、 SSH/CLI 设备驱动层设计 (Driver Layer)

为了应对多厂商（飞塔、华为、华三、山石等）的 CLI 差异，后端必须设计一个**抽象策略工厂**。由于大多数网络底层库（如 Netmiko）是同步阻塞的，必须使用 `asyncio.to_thread()` 将其扔进线程池，防止拖垮 FastAPI 的事件循环。

### 1. 驱动基类规范 (`BaseFirewallDriver`)

任何厂商的驱动类都必须实现以下三个核心抽象方法：

* `fetch_device_context()`: 上线拉取设备当前的地址对象、端口组及安全策略库，返回标准化 JSON 格式，供查重引擎计算 HASH。
* `build_object_commands(items)`: 根据查重结果（需要新创的对象），将 IP/端口组组装成该厂商的命令行（如华为的 `address-set`，飞塔的 `config firewall address`）。
* `build_rule_commands(items)`: 组装具体的安全策略行放行命令（如华三的 `security-policy ip` ➡️ `rule name` ），并注入计算出的 `source_zone` 和 `dest_zone`。

### 2. 边界墙 NAT 命令的组装铁律

当驱动解析到某个 `PushedPolicyItem` 对应的 `Policy` 带有 `source_snat_ip`（即当前墙是边界墙且需要 NAT）时：

* **对象层**：除了创建源/目的地址对象，驱动必须自动在防火墙上创建或引用对应的 **NAT 地址池对象**（基于 `source_snat_ip` 命名的 Pool）。
* **策略层**：驱动必须同时生成两条 CLI 命令块：
1. **安全策略放行命令**：放行 `源域 ➡️ 目的域`，源 IP 填 **转换前（原始）IP**，目的 IP 填真实目的 IP。
2. **NAT 策略转换命令**：指定 `源域 ➡️ 目的域`，源 IP 满足条件时，`action` 动作为 `source-nat`，转换为刚才创建的 **NAT 地址池对象**。

## 🛑 八、 异常管理、事务隔离与回滚策略

在企业级 NetDevOps 场景中，网络超时、凭据失效、设备 CPU 爆表是常态，后端必须具备极强的容错设计。

### 1. 实时流日志表的“独立事务隔离” (Crucial)

> **⚠️ 研发红线警告**：
> 在执行设备推送时，整个工单的多表更新通常在一个大的数据库 Session 事务中。如果中途发生异常导致大事务 `db.rollback()`，那么下发过程中的报错日志、历史流日志也会被无情地从数据库里擦除，导致运维人员根本看不见“刚才为什么失败”。

* **解决方案**：实现 `PushLog` 的**独立 Session 灌入机制**。每当下发引擎需要发出一条系统日志时，必须向数据库单独申请一个局部的、短生命周期的 `detached_session`，写入后立刻 `commit()` 释放。这样即使主进程崩溃回滚，完整的下发历史日志依然会完好无损地留在数据库中。

### 2. 熔断与动态回滚脚本生成

当倒序下发链条在中间某一台防火墙失联或报错时（例如目的端墙成功，但边界墙推送报错）：

* **熔断机制**：流水线立刻终止，不再向前推进（不再配置源端墙，防止灰度流量泄露）。
* **归档处理**：将当前墙的 Snapshot 置为 `failed`，将 `Order` 主表置为 `failed`。
* **回滚脚本留存**：系统不进行盲目的全自动反向物理删除（防止引发不可控的次生网络网络故障）。正确的做法是：**下发引擎利用驱动层，将已经成功推入的前序墙的 `raw_commands` 翻译成反向逆置删除脚本（例如将 `rule 10 name XX` 转换为 `undo rule 10`）**，将其回填存入 `PushedPolicySnapshot.error_log` 中，并在 UI 界面上提供一个 **“一键应急回滚”** 按钮，由运维人员人工确认后再点按执行。
